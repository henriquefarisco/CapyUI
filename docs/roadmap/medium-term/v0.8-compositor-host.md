# v0.8.0 — Compositor host + dirty rects

**Status:** delivered (CapyUI side, 2026-05-21). Lado CapyOS (compositor consumidor) segue trilha Etapa 4.
**ABI bump:** capy-ui-widget 0.6 → 0.8 (skip 0.7 — reservado para futuro adapter metadata bump quando Etapa 4 fechar)
**Display-list schema:** 2 → 3 (`CAPY_DL_DIRTY_HINT`)
**Dependência:** v0.6.0 entregue. Não bloqueada por Etapa CapyOS — opera apenas sobre o display-list.
**Estimativa:** M (CapyUI ✅) + L (CapyOS, pendente)
**Owner:** CapyUI

## Objetivo

Compositor CapyOS passa a consumir display-list com regiões sujas (dirty rects), permitindo redesenho parcial e melhor performance.

## Escopo CapyUI

- Op aditiva `CAPY_DL_DIRTY_HINT` no display-list (schema → 3):
  - `rect`: região suja explicitada.
  - Opcional; compositor pode ignorar e fazer diff próprio.
- API nova:
  ```c
  int capy_widget_diff(struct capy_display_list *prev,
                       struct capy_display_list *next,
                       struct capy_ui_rect *out_dirty,
                       uint32_t out_cap);
  ```
- `widget_diff` compara duas display-lists e produz lista de rects que mudaram.
- Algoritmo determinístico: mesma entrada → mesma saída byte-a-byte.

## Escopo CapyOS

- Implementação do compositor:
  - Surface manager com double-buffer.
  - Vsync coordenado com display driver.
  - Aplicação de dirty rects ao recompor.
- Sem mutação reversa do modelo CapyUI (compositor é consumidor read-only do display-list).

## Testes obrigatórios

1. **Diff de árvore idêntica** retorna 0 rects.
2. **Mudar 1 botão produz N rects pequenos**, não tela inteira.
3. **Animação contínua** não fragmenta dirty rects indefinidamente (rects coalescem quando próximos).
4. **Overflow de `out_cap`** retorna `-1` com `out_dirty` parcialmente preenchido até a capacidade.
5. **Determinismo:** mesma `(prev, next)` → mesma lista de rects.
6. **Display-list sem `DIRTY_HINT`** funciona normalmente (compositor faz diff próprio).

## Notas de compatibilidade

- Schema bump aditivo: consumidores antigos ignoram `DIRTY_HINT`.
- `widget_diff` é opcional; usar apenas se compositor não tem implementação própria.
- Performance: alvo é `diff` em O(n) onde n = max(count_prev, count_next).

## Objetivo

Alvos de performance mensuráveis e regressão guardada por CI. Pool allocator opcional e cache de measure.

## Escopo CapyUI

- Pool allocator opcional embutido:
  ```c
  struct capy_widget_pool {
    void *buffer;
    uint32_t size;
    uint32_t used;
    uint32_t high_water;
  };

  void capy_widget_pool_init(struct capy_widget_pool *p,
                             void *buf, uint32_t size);
  void capy_widget_pool_reset(struct capy_widget_pool *p);
  ```
- Wrapper allocator que usa pool:
  ```c
  struct capy_widget_allocator capy_widget_allocator_from_pool(
      struct capy_widget_pool *p);
  ```
- Campo aditivo em `capy_widget`: `uint8_t layout_dirty` + `uint32_t layout_version`.
- `capy_widget_measure` só recalcula sub-árvores com `layout_dirty=1` ou `layout_version` desatualizado.
- API `capy_widget_invalidate(struct capy_widget *w)` marca subárvore para recompute.
- Benchmark suite em `tests/bench/` (não bloqueia CI; registra métricas).

## Escopo CapyOS

Nenhum no core. O host pode escolher pool allocator para performance previsível.

## Testes obrigatórios

1. **Pool não-overflow** aloca múltiplos widgets sequencialmente.
2. **Pool overflow** retorna NULL sem corrupção.
3. **Reset libera tudo** mantendo `high_water` para diagnóstico.
4. **Cache de measure:** mutação isolada em subárvore não recomputa árvore inteira.
5. **`invalidate(w)`** marca ancestral path corretamente.
6. **Benchmark baseline:** 10000 widgets `measure+arrange+emit` documentado.

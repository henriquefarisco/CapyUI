## Escopo entregue (widget core)

Plumbing para resolução de ícones (sync) + geração de thumbnail (async). Widget core nunca decode PNG/JPEG, nunca toca disco — host implementa via callbacks + dispara `THUMBNAIL_READY` event quando pronto.

### Event type aditivo

`CAPY_WIDGET_EVENT_THUMBNAIL_READY` (appended ao tail do enum). Payload: `event->x = (int32_t)request_id`, `event->y = (int32_t)image_id`.

### Tail field em context

`struct capy_icon_provider *icon_provider` em `capy_widget_context` (NULL = fallback `image_id=0`).

### Struct + typedefs

```c
typedef int (*capy_icon_resolve_fn)(void *ud, const char *mime, const char *ext,
                                    uint32_t *out_icon_id);
typedef int (*capy_icon_thumbnail_request_fn)(void *ud, const char *path,
                                              uint32_t *out_request_id);

struct capy_icon_provider {
  capy_icon_resolve_fn               resolve;
  capy_icon_thumbnail_request_fn     thumbnail_request;
  void                              *user_data;
};
```

### 4 APIs

- `capy_widget_set_icon_provider(ctx, provider)` — attach/clear. NULL ctx no-op.
- `capy_icon_resolve(ctx, mime, ext, *out_id)` — forwards; **deterministic fallback `image_id=0`** quando: sem provider, callback NULL, ou host retorna -1. Returns 0 always (exceto NULL ctx/out_id).
- `capy_icon_thumbnail_request(ctx, path, *out_request_id)` — forwards; host devolve request_id ≥ 1. Returns -1 em NULL/missing callback/host failure/host devolve request_id=0.
- `capy_icon_thumbnail_ready(ctx, root, request_id, image_id)` — emite `THUMBNAIL_READY` em `root` (se non-NULL). Rejeita `request_id == 0` (reservado).

## Escopo desktop-session

File manager + desktop icons consomem `capy_icon_resolve` para ícones de mime/extension e `capy_icon_thumbnail_request` para previews de imagens/PDFs. Widget que solicitou o thumbnail recebe o `THUMBNAIL_READY` event e troca de placeholder para imagem real.

2. `icon_resolve_fallback_no_provider` — sem provider → `image_id=0`.
3. `icon_resolve_fallback_host_failure` — host -1 → fallback para 0.
4. `icon_thumbnail_request_forwards` — host atribui request_id incrementando.
5. `icon_thumbnail_request_failure_modes` — host -1 + host devolve 0 → ambos -1.
6. `icon_thumbnail_ready_rejects_id_zero` — request_id=0 rejeitado; outros OK.
7. `icon_no_provider_thumbnail_request_returns_negative` — sem provider → -1.
8. `icon_null_guards` — todas as combinações NULL.

## Decisões técnicas

- **Plumbing-only.** Widget core nunca decode bytes, nunca toca disco.
- **Fallback determinístico image_id=0.** Reservado para silhouette default; hosts atribuem `>= 1` para ícones reais.
- **request_id=0 reservado.** Hosts não podem usar como id válido — facilita "unassigned" checks.
- **Event in `_thumbnail_ready`.** Routing via `capy_widget_handle_event(root)` — não direto ao widget que pediu (dispatcher decide via x=request_id).
- **Stateless.** Widget core não rastreia request_id em flight — host responsável por cancelamento/timeout.

## Deferido para minors futuros

- **Icon theme cascade.** Múltiplos providers chained (user → desktop → fallback).
- **Vector icons.** SVG/PDF pipeline lives outside core.
- **Cancellation.** `capy_icon_thumbnail_cancel(request_id)`.
- **Stack/group folder icons.** v2.9 group folders precisa de `stack_count` no descriptor.

## Objetivo

Sistema unificado de ícones (UI + file types + apps) e thumbnails (imagens, vídeos, docs) com cache LRU, multi-DPI awareness e fallback determinístico. CapyUI provê o plumbing; o host decodifica e desenha.

## Escopo CapyUI

- Struct `capy_icon_descriptor`:
  ```c
  struct capy_icon_descriptor {
      const char *id;           /* "folder", "app.terminal", "mime.image-png" */
      uint16_t    base_size;    /* 16, 24, 32, 48, 64, 128, 256 */
      uint16_t    dpi_x256;     /* requisita variante para esse scale */
      uint8_t     theme_variant;/* 0=auto, 1=light, 2=dark, 3=hc */
      uint8_t     state;        /* normal, hover, active, disabled */
  };
  ```
- Callbacks host:
  ```c
  typedef int (*capy_icon_resolve_fn)(void *u,
      const struct capy_icon_descriptor *d, uint32_t *out_image_id);
  typedef int (*capy_thumbnail_request_fn)(void *u,
      const char *path, uint16_t want_w, uint16_t want_h,
      uint32_t *out_image_id);  /* async; -2 = pending */
  ```
- `struct capy_icon_provider` caller-owned + tail field `icon_provider` em `capy_widget_context`.
- 3 APIs core:
  - `capy_icon_resolve(ctx, descriptor, out_image_id)`.
  - `capy_thumbnail_request(ctx, path, w, h, out_image_id)`.
  - `capy_thumbnail_ready(ctx, image_id)` → 0=loading, 1=ready, -1=failed.
- Event aditivo `CAPY_WIDGET_EVENT_THUMBNAIL_READY` (host emite quando async completar; carrega `image_id` em `event->x`).
- Fallback determinístico: `out_image_id = 0` (default icon) quando resolve falha; nunca leak indeterminado.

## Escopo desktop-session

- File manager + desktop icons substituem image_ids hard-coded por `capy_icon_resolve` lookups.
- Thumbnails async em LIST/grid: enquanto `ready==0`, renderiza icon do MIME type; ao receber `THUMBNAIL_READY`, troca image_id e invalida o widget.
- Configuração: Settings → Appearance → "Show thumbnails for: images / videos / PDFs / never".

## Testes obrigatórios

1. **Resolve** retorna image_id do mock determinístico.
2. **Fallback** em resolve fail → image_id=0.
3. **Thumbnail request** sync no mock retorna ready=1 imediatamente.
4. **Thumbnail pending** → ready=0; após event THUMBNAIL_READY → ready=1.
5. **DPI scaling**: descriptor com `dpi_x256=512` requisita variante 2×.
6. **Theme variant auto** segue o tema ativo do contexto.
7. **NULL guards** + descriptor inválido (base_size=0) → -1.

## Notas

- CapyUI não decodifica PNG/JPEG/SVG; host responsável. CapyUI armazena só `image_id` (uint32) que o backend resolve.
- Cache LRU vive no host; CapyUI não tem política.
- Icon ids canônicos documentados em `docs/roadmap/contracts/icon-ids.md` (novo).

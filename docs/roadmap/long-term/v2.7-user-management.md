## Escopo entregue (widget core)

- 2 macros: `CAPY_USER_NAME_MAX` (=32), `CAPY_USER_DISPLAY_NAME_MAX` (=64).
- `struct capy_user_account { char username[32]; char display_name[64]; uint32_t uid; uint8_t is_admin; uint8_t is_locked; uint16_t avatar_image_id; uint32_t last_login_ms; }`.
- 5 typedefs (callbacks no host):
  - `capy_user_list_fn(user_data, out, cap) → int` — modos enumerados.
  - `capy_user_create_fn(user_data, *account, password) → int` — host valida/hashea; CapyUI nunca armazena password.
  - `capy_user_update_fn(user_data, *account) → int` — lookup por `uid`.
  - `capy_user_delete_fn(user_data, uid) → int`.
  - `capy_user_set_avatar_fn(user_data, uid, png, len) → int`.
- `struct capy_user_directory { list, create, update, del, set_avatar, user_data }` (caller-owned).
- Tail field `user_directory` em `capy_widget_context` (NULL = surface ausente).
- 6 APIs:
  - `capy_widget_set_user_directory(ctx, dir)` — attach/clear. NULL ctx no-op.
  - `capy_user_list(ctx, out, cap)` — forwards; `-1` em NULL/missing callback ou `cap>0 && out==NULL`.
  - `capy_user_create(ctx, account, password)` — valida `username[0] != '\0'` antes de chamar host; password é forwardada verbatim.
  - `capy_user_update(ctx, account)` — forwards; host faz lookup por uid.
  - `capy_user_delete(ctx, uid)` — rejeita `uid == 0` (root) antes do host.
  - `capy_user_set_avatar(ctx, uid, png, len)` — rejeita `len > 0 && png == NULL`; `len == 0 && png == NULL` é "clear avatar".

## Escopo desktop-session

App **Settings → Users** (deferido para fatia desktop-session subsequente; ABI já habilita). Lista (LIST widget v0.x), formulário (TEXTBOX), botões add/edit/remove/avatar. Confirmação destrutiva no remove. Apenas admins veem botões mutativos.

## Testes entregues (7)

1. `user_list_forwards_to_host` — roster de 2 conta enumerado; uid/is_admin preservados.
2. `user_create_appends_and_passes_password` — host recebe password verbatim; conta added.
3. `user_create_rejects_empty_username` — `username[0] == '\0'` → -1 antes do host.
4. `user_update_roundtrip` — flag `is_admin`/`is_locked` toggleadas via update.
5. `user_delete_rejects_root_uid_zero` — uid=0 rejeitado; uid normal passa.
6. `user_set_avatar_clear_and_set` — set com bytes, clear com `(NULL, 0)`, `(NULL, >0)` rejeitado.
7. `user_no_directory_returns_negative` — sem `set_user_directory`, todas as 5 APIs retornam -1; setter NULL ctx é no-op.

## Notas

- **CapyUI nunca armazena senha.** `password` em `create_fn` é forward direto ao host.
- **Composição:** v2.13 login screen consome a mesma `user_directory` para listar avatares + autenticar.
- **`uid == 0` é root** — undeletable pelo widget core. Host pode redefinir convenções via wrapper.

## Objetivo

App Settings → Users que permite listar, criar, editar, remover e trocar avatar de contas. CapyUI fornece a UI; o host CapyOS implementa o backend real (autenticação, persistência, permissões) via callbacks.

## Escopo CapyUI

- Struct `capy_user_account` em `src/widget/capy_widget.h`:
  ```c
  struct capy_user_account {
      char     username[32];
      char     display_name[64];
      uint32_t uid;
      uint8_t  is_admin;
      uint8_t  is_locked;
      uint16_t avatar_image_id;   /* host-owned, 0 = default */
      uint32_t last_login_ms;
  };
  ```
- Callbacks no host:
  ```c
  typedef int (*capy_user_list_fn)(void *u, struct capy_user_account *out, uint32_t cap);
  typedef int (*capy_user_create_fn)(void *u, const struct capy_user_account *a, const char *password);
  typedef int (*capy_user_update_fn)(void *u, const struct capy_user_account *a);
  typedef int (*capy_user_delete_fn)(void *u, uint32_t uid);
  typedef int (*capy_user_set_avatar_fn)(void *u, uint32_t uid, const void *png, uint32_t len);
  ```
- `struct capy_user_directory` caller-owned + tail field `user_directory` em `capy_widget_context`.
- 5 APIs forwarding no core (`capy_user_list/create/update/delete/set_avatar`).

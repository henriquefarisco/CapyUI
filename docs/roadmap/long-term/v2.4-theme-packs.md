## Objetivo

Bundle binário compacto de tokens de tema para distribuição offline. Sucessor evoluído do `capy_theme_serialize` textual de v0.14: o pack é binário canonical LE, fail-closed em magic/version/checksum/reserved, idempotente e zero-alloc.

## Escopo entregue

Formato binário `.captheme` (magic `"CTHM"`, format_version=1):

```
offset  size  field
0       4     magic = "CTHM"
4       2     format_version (uint16)
6       1     variant (CAPY_THEME_VARIANT_*; rejected if >= 16)
7       1     high_contrast (0 ou 1)
8       4     FNV-1a 32-bit checksum sobre bytes [16, end)
12      2     entry_count (uint16)
14      2     reserved (must be 0)
16+     ...   N entradas de 6 bytes:
                uint8  token_id (>=1, <CAPY_TOKEN_COUNT, senão skipped)
                uint8  reserved (must be 0)
                uint32 colour_le (0xAARRGGBB little-endian)
```

### APIs

```c
int capy_theme_pack_validate(const void *buf, uint32_t len);
int capy_theme_pack_load(const void *buf, uint32_t len,
                         struct capy_theme *out);
```

- `_validate` checa magic, format_version ≤ `CAPY_THEME_PACK_FORMAT_VERSION`, reserved zero, `len == header + count*entry`, e FNV-1a checksum.
- `_load` valida + aplica tokens em `*out` (caller seeds com `default_light/dark/high_contrast` antes). Tokens ausentes ficam intactos; tokens com id >= `CAPY_TOKEN_COUNT` são silenciosamente skipados (forward-compat).
- `out->variant` e `out->high_contrast` são sobrescritos; `out->version` preservado para versionamento per-instance do host.

### Macros publicadas

`CAPY_THEME_PACK_MAGIC0..3`, `CAPY_THEME_PACK_FORMAT_VERSION` (=1), `CAPY_THEME_PACK_HEADER_SIZE` (=16), `CAPY_THEME_PACK_ENTRY_SIZE` (=6).

## Testes entregues (8)

1. `validate_well_formed_zero` — pack válido + NULL/zero guards.
2. `load_applies_tokens` — tokens overwritten, variant/high_contrast aplicados, version preservado.
3. `bad_magic_rejected` — magic alterada → -1 em validate e load.
4. `future_version_rejected` — format_version+1 → -1 mesmo com checksum recomputado.
5. `checksum_mismatch_rejected` — flip de byte no body sem re-checksum.
6. `unknown_token_skipped` — entry com token_id=250 silently skipada; entry com token_id válido aplicada.
7. `truncated_buffer_rejected` — declared count > buffer; header só; abaixo de header.
8. `reserved_field_nonzero_rejected` — offset 14 != 0 mesmo com checksum válido.

## Deferido para minors futuros

- **Hot reload.** Hosts observam arquivo e re-chamam `_load`. CapyUI não polla.
- **Assinatura (Ed25519).** Hosts validam externamente. Formato 2 (futuro) pode anexar bloco trailing.
- **Refs de fonts.** `font_id` vive no widget desde v0.9; pack 2.4 carrega só cores.
- **Refs de ícones.** Surface v2.11 (icon descriptor + provider).
- **Metadata.** name/author/locale ficam fora do core; hosts wrap o pack num envelope que carrega esses campos.

## Notas

- **Decoupling preservado.** Pack vive em buffer caller-owned; CapyUI nunca abre arquivo, nunca aloca.
- **Determinismo.** Mesmo input → mesmo `*out`. Endianness explícita.
- **Compatível com `capy_theme_serialize` (v0.14):** os formatos coexistem. Textual key=value para diff/git; binário para distribution.

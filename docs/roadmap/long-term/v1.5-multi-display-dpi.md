# v1.5.0 — Multi-display e DPI scaling

**Status:** delivered (CapyUI side, 2026-05-21) com escopo conservador. Layout auto-scaling profundo em `capy_widget_measure`/`arrange` fica para fatia futura aditiva; em 1.5 a integração é via helper opt-in + marker no display-list.
**ABI bump:** capy-ui-widget 1.4 → 1.5 (aditivo)
**Display-list schema:** 4 → **5** (`CAPY_DL_DPI_SCOPE`). Primeiro bump real de schema desde a 0.9; o slice doc original projetava 5→6 porque assumia que slices anteriores teriam bumpado, mas 1.0–1.4 mantiveram schema em 4.
**Dependência:** v1.4.0 entregue
**Estimativa:** L (CapyUI ✅)
**Owner:** CapyUI

## Objetivo

Suporte a múltiplos displays com DPI diferentes (1x, 1.5x, 2x, 3x). Layout fluido; snap em grid de pixel inteiro.

## Escopo CapyUI

- API:
  ```c
  void capy_widget_set_dpi_scale(struct capy_widget_context *ctx,
                                 uint16_t scale_x256);
  ```
  - `scale_x256 = 256` → 1.0× (default).
  - `scale_x256 = 384` → 1.5×.
  - `scale_x256 = 512` → 2.0×.
- Layout multiplica `padding`, `gap`, `min_w/h`, `max_w/h` por `scale_x256 / 256` (em fixed-point).
- Snap em pixel inteiro após escala (evita sub-pixels).
- Op aditiva `CAPY_DL_DPI_SCOPE` no display-list (schema → 6):
  - `rect`: região afetada.
  - `font_size`: ajustado para o DPI.
  - Compositor honra ou ignora conforme suporte.
- API:
  ```c
  uint16_t capy_widget_get_dpi_scale(const struct capy_widget_context *ctx);
  ```

## Testes que cobrem o contrato

- `test_dpi_default_is_256` — `capy_widget_context_init` seeds 256; getter retorna 256.
- `test_dpi_set_get_roundtrip` — set 384/512 roundtrip; zero clampa para 1; NULL guards (set no-op; get retorna 256).
- `test_dpi_scale_dim_helper` — 1× identidade; 2× dobra (positive/negative); 1.5× trunca toward zero; 3× triplica; saturating clamp em `INT32_MAX/MIN`; scale=0 retorna 0.
- `test_dpi_scope_op_emitted_when_non_default` — `dl.dpi_scale_x256 = 384` → primeiro op é `CAPY_DL_DPI_SCOPE` com `image_id=384` e `rect=root.bounds`; segundo é o CLIP_PUSH usual.
- `test_dpi_scope_op_omitted_at_default` — `dl.dpi_scale_x256 == 256` (default) → nenhuma op DPI_SCOPE no display-list (preserva byte-equivalência com pre-1.5).
- `test_dpi_independent_contexts` — dois contextos com scales diferentes (384 e 512) mantêm estado independente.
- `test_dpi_gpu_translator_skips_dpi_scope` — DPI_SCOPE + RECT no DL → `capy_dl_to_quads` emite exatamente 1 quad (para o RECT); não falha em DPI_SCOPE.

**Não testado em 1.5** (layout auto-scale): o slice doc original previa testes para `padding`/`gap`/`min_w`/`max_w` escalados automaticamente em `measure`/`arrange`. Esses testes ficam para a fatia futura que implementar o auto-scaling; em 1.5 o helper `capy_widget_dpi_scale_dim` permite que código do host realize o scaling de forma determinística.

## Notas

- Aditivo. Hosts sem multi-DPI usam scale 256 (default) e não veem nenhuma mudança comportamental.
- Compositor CapyOS lê `DPI_SCOPE` para escolher rasterizer adequado; o op carrega o scale em `image_id` (reuso aditivo do slot 32-bit existente, sem aumentar `sizeof(capy_dl_cmd)`).
- Fontes (v0.9 metrics hook): em 1.5 o scale aplica-se uniformemente; o host pode passar `dpi_scale_dim(scale, font_size)` para a métrica hook se quiser tipografía escalada. Split futuro (font scale separado do dimension scale) fica como evolução aditiva via `reserved_dpi` no `capy_display_list`.
- Layout auto-scale deferido: a invariante de byte-equivalência com pre-1.5 a `scale=256` foi privilegiada para evitar cascata em 125 testes existentes. Quando o auto-scale vier, ele será controlado por uma flag/op-in para preservar essa invariante para callers que dependem de bounds exatos.

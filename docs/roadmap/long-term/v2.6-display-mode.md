# v2.6.0 — Display mode (resolution + refresh rate)

**Status:** delivered (2026-05-21)
**ABI bump:** capy-ui-widget 2.5 → 2.6 (aditivo) + capy-ui-desktop-session 1 (aditivo, sem bump)
**Display-list schema:** 7 (sem mudança)
**Dependência:** v1.5 (DPI scaling) entregue
**Estimativa:** M
**Owner:** —

## Escopo entregue (widget core)

- `#define CAPY_DISPLAY_MODE_FLAG_PREFERRED 0x01u`, `CAPY_DISPLAY_MODE_FLAG_INTERLACED 0x02u`.
- `struct capy_display_mode { uint16_t width, height, refresh_hz_q8; uint8_t bpp, flags; }`.
- Typedefs `capy_display_enum_modes_fn(user_data, out, cap) → int`, `capy_display_set_mode_fn(user_data, *mode) → int`.
- `struct capy_display_controller { enum_modes, set_mode, user_data, current_mode, has_current, reserved[3] }` (caller-owned).
- Tail field `display_controller` em `capy_widget_context` (NULL = surface ausente).
- 4 APIs novas:
  - `capy_widget_set_display_controller(ctx, controller)` — attach/clear (NULL = clear).
  - `capy_display_enum_modes(ctx, out, cap)` — forwards a `controller->enum_modes`; retorna count, -1 em NULL/missing callback ou `cap>0 && out==NULL`.
  - `capy_display_set_mode(ctx, root, mode)` — valida `width>0 && height>0`, forwards a `controller->set_mode`; em sucesso cacheia `current_mode`, set `has_current=1` e emite `CAPY_WIDGET_EVENT_DISPLAY_MODE_CHANGED` em `root` (se non-NULL). Payload: `event->x = (width<<16)|height`, `event->y = (bpp<<16)|refresh_hz_q8`.
  - `capy_display_current_mode(ctx, out)` — copia `current_mode` se `has_current`; -1 caso contrário.
- Event type aditivo `CAPY_WIDGET_EVENT_DISPLAY_MODE_CHANGED` (appended ao tail do enum).

## Escopo desktop-session

App **Settings → Display** (deferido para fatia desktop-session subsequente; ABI já habilita). Lista modos enumerados, mostra atual destacado, botão "Apply" + rollback "Revert in 15s". Persistência do último mode aplicado via `capy_widget_serialize` (1.10) num blob `/var/state/capyui/display.bin` é responsabilidade do host.

## Testes entregues (7)

1. `display_enum_modes_forwards_to_host` — catalog de 3 modos enumerado; preferred flag preservada.
2. `display_set_mode_caches_and_emits_event` — set bem-sucedido cacheia `current_mode` + `has_current=1`; current_mode legível depois.
3. `display_set_mode_host_failure_no_cache` — host callback retorna -1 → `has_current` permanece 0.
4. `display_no_controller_returns_negative` — sem `set_display_controller`, todas as 3 APIs retornam -1.
5. `display_set_mode_rejects_zero_dimensions` — width=0 ou height=0 → -1 antes mesmo de chamar host.
6. `display_enum_modes_capacity_truncates` — cap < count → host trunca; CapyUI repassa.
7. `display_null_guards` — NULL ctx → -1 em todas; setter NULL ctx é no-op.

## Deferido para minors futuros

- **Fractional refresh.** Promover `refresh_hz_q8` para Q24.8 quando 23.976 / 59.94 / 119.88 Hz / G-SYNC precisarem representação exata.
- **HDR + VRR metadata.** Flags reservados para HDR10, Dolby Vision, VRR range, color gamut.
- **Rollback/preview UI 15s.** Vive no desktop-session, não no widget core.
- **Persistência por usuário.** Via state serialization 1.10 + envelope do host.

## Objetivo

Permitir que o host enumere modos de vídeo suportados (resolução × refresh) e que a sessão de desktop selecione um modo em runtime via painel de configurações. Caller-buffer-backed; sem alocação no core.

## Escopo CapyUI

- Struct nova em `src/widget/capy_widget.h`:
  ```c
  struct capy_display_mode {
      uint16_t width;
      uint16_t height;
      uint16_t refresh_hz_q8;   /* Q8.0; 60 = 60Hz, 144 = 144Hz, 240 = 240Hz */
      uint8_t  bpp;             /* 24 ou 32 */
      uint8_t  flags;           /* CAPY_DISPLAY_MODE_PREFERRED, _INTERLACED */
  };
  ```
- Callbacks no host (típicos backends framebuffer/DRM/CapyOS adapter):
  ```c
  typedef int (*capy_display_enum_modes_fn)(void *user_data,
      struct capy_display_mode *out, uint32_t cap);
  typedef int (*capy_display_set_mode_fn)(void *user_data,
      const struct capy_display_mode *mode);
  ```
- Struct `capy_display_controller` (caller-owned) atachada ao `capy_widget_context` por tail field aditivo `display_controller`.
- 3 APIs no core:
  - `capy_display_enum_modes(ctx, out, cap)` → forwards.
  - `capy_display_set_mode(ctx, mode)` → forwards + emit do event `CAPY_WIDGET_EVENT_DISPLAY_MODE_CHANGED`.
  - `capy_display_current_mode(ctx, out)` → cache do último mode aplicado.
- Event type novo appended ao tail do enum (`DISPLAY_MODE_CHANGED`).

## Escopo desktop-session

- App **Settings → Display**: lista modos enumerados, mostra atual destacado, botão "Apply" + "Revert in 15s" (rollback se usuário não confirmar).
- Persistência: último mode confirmado salvo via `capy_widget_serialize` (1.10) num blob em `/var/state/capyui/display.bin`.

## Testes obrigatórios

1. **Enum modes** retorna lista determinística do user_data mock.
2. **Set mode** propaga ao callback e emite `DISPLAY_MODE_CHANGED`.
3. **Current mode** cacheado entre chamadas.
4. **NULL guards** em ctx/controller/out → -1.
5. **Capacity zero** em enum → returna count sem escrever.
6. **Refresh Q8** preserva 60/75/120/144/240Hz sem perda.
7. **Persist round-trip** desktop-session restaura último mode.

## Notas

- CapyUI não toca framebuffer real; só plumbing. Backend CapyOS aplica o mode e re-emite damage full.
- Modos não enumerados pelo host nunca aparecem na UI — fail-closed.
- `flags` reservado para extensões futuras (HDR, variable refresh).

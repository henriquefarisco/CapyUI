# v2.14.0 — Advanced widget state: date picker

**Status:** delivered (commit-side; gates externos pendentes como toda tag)
**ABI bump:** capy-ui-widget 2.13 → **2.14 (aditivo)**
**Display-list schema:** permanece **7**
**Dependência:** v2.1 (advanced widget enums) entregue
**Estimativa:** S
**Owner:** —

## Contexto

Os 8 "advanced widgets" entregues em **2.1** são, até aqui, **enum-only**:
`TREE/TABLE/RICH_TEXT/CANVAS/CHART/COLOR_PICKER/DATE_PICKER/AUTOCOMPLETE`
existem como slots no `enum capy_widget_type` e integram com o pipeline
genérico (layout, emit, focus, serialize, a11y), mas **não têm estado nem
APIs dedicadas** — o comentário do enum em `capy_widget.h` diz explicitamente
que o estado "lives in caller-owned structures referenced via `user_data`
until a dedicated 2.x minor lifts each into a tail-field slot".

Esta fatia é o **primeiro desses minors dedicados**: levanta o `DATE_PICKER`
de `user_data` para um campo de cauda de primeira classe, estabelecendo o
padrão aditivo que as fatias seguintes (TABLE columns, TREE hierarchy,
COLOR_PICKER value, CHART datasets, AUTOCOMPLETE suggestions, ...) vão repetir.

## Escopo CapyUI (entregue)

### 1. Tipo de valor

```c
struct capy_date {
  uint16_t year;  /* 1..65535; 0 = unset */
  uint8_t month;  /* 1..12; 0 = unset */
  uint8_t day;    /* 1..31, calendar-validated; 0 = unset */
};
```

### 2. Campo de cauda

`struct capy_date date_value;` appended ao tail de `struct capy_widget`
(válido só quando `type == CAPY_WIDGET_DATE_PICKER`). `capy_widget_create`
zera tudo → sentinela "unset". `sizeof(struct capy_widget)` cresce 4 bytes
(crescimento de cauda permitido, igual a 1.9/2.0/2.2).

### 3. APIs (total, determinístico, zero-alloc, zero-float)

- `int capy_date_is_valid(uint16_t year, uint8_t month, uint8_t day)` —
  predicado de calendário Gregoriano proléptico puro (mês 1..12, dia dentro
  do mês incluindo 29/fev em ano bissexto, ano ≥ 1). Regra de bissexto:
  `(year%4==0 && year%100!=0) || year%400==0`.
- `int capy_widget_set_date(w, year, month, day)` — valida e armazena. `0` em
  sucesso; `-1` em NULL / `type != DATE_PICKER` / data inválida. **Fail-closed:**
  o valor armazenado não muda em falha.
- `int capy_widget_clear_date(w)` — reseta para a sentinela unset. `0`; `-1`
  em NULL / tipo errado.
- `int capy_widget_get_date(w, out)` — copia o valor em `*out`. Retorna `1`
  quando há data válida, `0` quando unset, `-1` em NULL / tipo errado /
  `out == NULL`.

DL schema permanece 7: o DATE_PICKER continua emitindo pelo pipeline genérico;
esta fatia só adiciona estado + validação.

## Testes obrigatórios (entregues — 8)

1. `date_set_valid_and_get` — 2024/02/29 (bissexto) seta e lê de volta.
2. `date_unset_get_returns_zero` — widget novo retorna 0 (unset).
3. `date_invalid_rejected_fail_closed` — datas inválidas → -1, valor anterior intacto.
4. `date_clear_resets` — set → clear → get retorna 0.
5. `date_wrong_widget_type_rejected` — set/clear/get em BUTTON → -1.
6. `date_is_valid_calendar_predicate` — fronteiras bissexto (2000/1900/2024) + bounds de mês/dia.
7. `date_null_guards` — NULL widget e NULL out → -1.
8. `date_determinism` — mesma escrita → mesma leitura repetida.

Total da suíte: **273 → 281** (guard de contagem pinado em `make version-check`).

## Compatibilidade

- 100% aditivo. Callers que mantinham a data em `user_data` pré-2.14 não são
  afetados — o campo fica unset até o setter ser usado.
- Macros bumpadas para `2/14/0`; `CAPYUI_API_VERSION_TAG = 0x00020E00`.
- `capy_widget_serialize` (1.10) **não** serializa `date_value` (continua na
  lista de campos deferidos); o formato binário não muda.

## Deferido para fatias seguintes

- Demais famílias advanced-widget (TABLE columns, TREE hierarchy,
  COLOR_PICKER value, CHART datasets, AUTOCOMPLETE suggestions, RICH_TEXT
  ranges, CANVAS draw callback) — cada uma um minor 2.x focado no mesmo padrão.
- Formatação/locale da data (fica no host / desktop-session via `capy_locale`).
- Serialização do `date_value` no estado (1.10) — minor aditivo futuro.

# v1.8.0 — IME rico

**Status:** delivered (CapyUI side, 2026-05-21). Modelo IME completo (preedit + candidate list + composition phase) entregue no widget core; engine real fica para o host CapyOS (mozc/anthy/hangul-engine).
**ABI bump:** capy-ui-widget 1.7 → 1.8 (aditivo)
**Display-list schema:** permanece **5** (candidate window deferida para fatia futura).
**Dependência:** v1.7.0 entregue + v0.4 (text edit)
**Estimativa:** L (CapyUI ✅)
**Owner:** CapyUI

## Objetivo

Modelo IME completo: candidate window, composition state expandido, contratos para CJK/Hangul/Indic. Host fornece engine real.

## Escopo CapyUI

- Struct:
  ```c
  struct capy_ime_state {
    const char *preedit;       /* texto em composição */
    uint16_t preedit_len;
    uint16_t cursor_pos;       /* posição no preedit */
    const char **candidates;   /* lista de candidatos */
    uint16_t candidate_count;
    uint16_t selected_index;
    uint8_t composition_phase; /* 0=idle, 1=composing, 2=selecting */
    uint8_t reserved[3];
  };
  ```
- APIs:
  - `void capy_ime_set_preedit(struct capy_widget *w, const char *text, uint16_t len, uint16_t cursor);`
  - `void capy_ime_set_candidates(struct capy_widget *w, const char **list, uint16_t count, uint16_t selected);`
  - `void capy_ime_commit(struct capy_widget *w, const char *final_text, uint16_t len);`
  - `void capy_ime_cancel(struct capy_widget *w);`
- Eventos abstratos:
  - `CAPY_WIDGET_EVENT_IME_COMPOSE_START`
  - `CAPY_WIDGET_EVENT_IME_PREEDIT_UPDATE`
  - `CAPY_WIDGET_EVENT_IME_COMMIT`
  - `CAPY_WIDGET_EVENT_IME_CANCEL`
- Candidate window renderizada via display-list (pequeno popup com lista de candidatos).

## Escopo CapyOS

- Engine IME real (mozc, anthy, Hangul-engine, ou subset próprio).
- Bridge eventos do IME → APIs CapyUI acima.

## Testes que cobrem o contrato

- `test_ime_set_preedit_marks_composing` — preedit "你" + cursor=3 setam fields e `composition_phase=1`; NULL text limpa preedit fields.
- `test_ime_set_candidates_marks_selecting` — lista de 3 candidatos com selected=1 → `composition_phase=2`; count=0 clear candidates mas preserva phase; NULL list também clear.
- `test_ime_commit_inserts_and_clears` — commit insere "hi" no textbox (via `capy_textbox_insert`) e limpa todo o state IME.
- `test_ime_cancel_preserves_textbox_content` — set_preedit em textbox com "abc"; cancel preserva "abc" e limpa IME.
- `test_ime_apis_are_noop_on_non_textbox` — BUTTON ignora set_preedit/set_candidates/cancel; commit retorna -1.
- `test_ime_event_types_routed_without_crash` — 4 IME event types são no-ops no core (host dispatcher é quem chama as APIs).
- `test_ime_get_state_snapshot` — snapshot via `capy_ime_get_state` reflete fields do widget; non-textbox e NULL widget zero-fill; NULL out é no-op.

**Não testado em 1.8** (deferido):
- Candidate window rendering via display-list (op aditiva futura, ex. `CAPY_DL_CANDIDATE_WINDOW`). Em 1.8 o host renderiza via desktop session sobre o `capy_ime_get_state` snapshot.
- Hangul jamo combination end-to-end — é trabalho do engine real (mozc/hangul-engine), não do widget core.
- Candidate selection via setas — dispatcher do host atualiza `selected_index` via `capy_ime_set_candidates` em resposta a KEY_DOWN.

## Notas

- Modelo IME no core; engine real no host. CapyUI nunca aloca/libera as strings de `preedit`, `final_text` nem o array de `candidates` — o host engine mantém alive até a próxima chamada `capy_ime_*`.
- v0.4 IME stub `capy_widget_ime_compose` continua presente para callers que só precisam de preedit forwarding mínimo sem candidate window. O modelo de 1.8 é strictly aditivo.
- 4 event types abstratos (IME_COMPOSE_START/PREEDIT_UPDATE/COMMIT/CANCEL) são pass-through no core. Host dispatcher os usa para sinalizar redraw/notificações sem couplar diretamente aos setters.
- Candidate window no display-list **adiada**: extender aditivamente via `CAPY_DL_CANDIDATE_WINDOW` (DL schema 5 → ?). Em 1.8 o host renderiza a popup via desktop session lendo `capy_ime_get_state`.
- Determinístico: dada uma sequência de chamadas `(set_preedit, set_candidates, commit|cancel)`, o estado final do widget é sempre o mesmo.

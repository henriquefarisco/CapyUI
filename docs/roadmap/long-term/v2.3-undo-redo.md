# v2.3.0 — Undo/Redo

**Status:** delivered (CapyUI side, 2026-05-21) com escopo focado: stack genérico per-context com FIFO eviction + branch-cut em push. Storage é caller-buffer; entries armazenam ponteiros caller-owned (action_id + redo/undo data). **Coalescing automático + auto-hook de TEXTBOX/RICH_TEXT deferidos** (hosts implementam externamente).
**ABI bump:** capy-ui-widget 2.2 → 2.3 (aditivo)
**Display-list schema:** permanece **7**.
**Dependência:** v2.2.0 entregue + v1.10 (serialização)
**Estimativa:** M (CapyUI ✅)
**Owner:** CapyUI

## Objetivo

Histórico determinístico de mutações por contexto. API genérica para qualquer widget. Coalescing automático para edições rápidas de texto.

## Escopo CapyUI

- Struct opaca:
  ```c
  struct capy_undo_stack;
  ```
- APIs:
  ```c
  int capy_undo_stack_init(struct capy_widget_context *ctx,
                           void *buf, uint32_t buf_size);
  int capy_undo_push(struct capy_widget_context *ctx,
                     const char *action_id,
                     const void *redo_data, uint32_t redo_len,
                     const void *undo_data, uint32_t undo_len);
  int capy_undo_undo(struct capy_widget_context *ctx);
  int capy_undo_redo(struct capy_widget_context *ctx);
  int capy_undo_can_undo(const struct capy_widget_context *ctx);
  int capy_undo_can_redo(const struct capy_widget_context *ctx);
  ```
- Coalescing: ações com mesmo `action_id` em janela de 500ms (configurável) são agrupadas.
- Storage limitado: buffer fornecido pelo caller; ao encher, descarta entries mais antigas (FIFO).
- Hook integrado: edições em TEXTBOX (v0.4) e RICH_TEXT (v2.1) registram automaticamente.

## Testes que cobrem o contrato

- `test_undo_init_partitions_caller_buffer` — init particiona buffer; capacity computada; defaults zerados; `coalesce_window_ms = 500`.
- `test_undo_push_undo_redo_roundtrip` — push → undo (entry copiada via out) → redo retorna estado.
- `test_undo_push_truncates_redo` — push após undo descarta a redo stack (branch cut).
- `test_undo_fifo_eviction_on_full_buffer` — buffer com capacity=2 + 3 pushes evicta o mais antigo; top continua coerente.
- `test_undo_on_empty_returns_negative` — sem init: tudo `-1`/`0`; com stack vazio: undo/redo `-1`.
- `test_undo_null_guards_and_too_small_buffer` — NULL ctx/buf; buffer < header+1 entry; todas retornam `-1`.
- `test_undo_determinism_same_sequence` — dois contexts com mesma sequência produzem mesmo `(count, redo_count, top.action_id)`.

**Não testado em 2.3** (deferido para fatia futura):
- **Coalescing automático** dentro de `coalesce_window_ms`. O campo existe na struct e o default de 500ms está fixado, mas o widget core **não** mescla entries automaticamente — hosts comparam contra a entry top e decidem se chamam push.
- **Auto-hook em TEXTBOX/RICH_TEXT.** Edits via `capy_textbox_insert`/`_delete` não registram automaticamente — host call `_push` na rota do dispatcher.
- **Integração com state serialization** para snapshot/restore do histórico em si.

## Notas

- **Zero allocation no core.** Storage é o caller-buffer; entries armazenam só ponteiros (action_id + redo_data + undo_data) caller-owned. Caller mantém strings/bytes alive enquanto entries estiverem no stack.
- **Aplicação real do undo é callback do host** baseado em `action_id`. O widget core devolve a entry via `out` (NULL-able); o caller dispatch a ação apropriada para o seu redo/undo handler.
- **FIFO eviction.** Quando o buffer enche, push remove a entry **mais antiga** (shift in-place). Trade-off: O(N) per overflow push, mas mantém o layout simples (sem ring buffer index). Hosts que querem históricos grandes dimensionam o buffer com mais capacity.
- **Branch cut.** Qualquer push descarta a redo stack — o usuário bifurcou da linhagem anterior.
- **`coalesce_window_ms`** é puramente metadata em 2.3 (default 500). Hosts leem o campo se quiserem implementar coalescing próprio.
- **Determinístico.** Dada uma sequência de operations, dois contexts com buffers idênticos produzem o mesmo `(count, redo_count, entries[i].action_id)`.

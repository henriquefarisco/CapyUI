# v2.2.0 — Virtualização

**Status:** delivered (CapyUI side, 2026-05-21) com **escopo conservador**: data source plumbing (`struct capy_virtual_data_source`, tail field `virtual_source` em `capy_widget`, 3 APIs `set_virtual_source`/`virtual_count`/`virtual_get_item`, event type `CAPY_WIDGET_EVENT_VIRTUAL_REQUEST_RANGE`). O **viewport-only emit rewiring** dos LIST/TREE/TABLE renderers + row recycling ficam para uma fatia futura aditiva que toca `capy_widget_emit_recursive` — a infraestrutura para isso está toda no lugar.
**ABI bump:** capy-ui-widget 2.1 → 2.2 (aditivo)
**Display-list schema:** permanece **7** (virtualization não adiciona ops; usa o pipeline genérico).
**Dependência:** v2.1.0 entregue
**Estimativa:** L (CapyUI ✅ — plumbing entregue; viewport emit é follow-up)
**Owner:** CapyUI

## Objetivo

LIST, TREE e TABLE renderizam apenas itens visíveis (viewport). Suporte a milhões de itens sem regressão de frame budget.

## Escopo CapyUI

- Conceito de **virtual data source**:
  ```c
  typedef int (*capy_virtual_get_count_fn)(void *user_data);
  typedef int (*capy_virtual_get_item_fn)(void *user_data, uint32_t index,
                                          char *out_text, uint16_t cap);

  struct capy_virtual_data_source {
    capy_virtual_get_count_fn get_count;
    capy_virtual_get_item_fn get_item;
    void *user_data;
  };

  void capy_widget_set_virtual_source(struct capy_widget *w,
                                      const struct capy_virtual_data_source *src);
  ```
- LIST/TREE/TABLE consultam source apenas para items no viewport (calculado a partir de scroll position + bounds).
- Recycle de child widgets: pool fixo de renderable rows; reusados ao scrollar.
- Eventos: `CAPY_WIDGET_EVENT_VIRTUAL_REQUEST_RANGE(start, end)` permite host pre-fetch.

## Testes que cobrem o contrato (plumbing)

- `test_virtual_set_source_stores_pointer` — set + clear + NULL widget guard.
- `test_virtual_count_forwards_to_callback` — `virtual_count` chama `get_count(user_data)` uma vez e retorna 42.
- `test_virtual_get_item_decodes_index` — emite "0", "42", "999"; out-of-range index retorna -1.
- `test_virtual_null_source_returns_negative` — sem source attach: helpers retornam -1; source com fn ptrs NULL: retorna -1.
- `test_virtual_helpers_null_guards` — NULL widget, NULL out_text, cap=0; callback nunca é chamado.
- `test_virtual_request_range_event_routes_without_crash` — event type pass-through em `capy_widget_handle_event`.
- `test_virtual_source_on_non_list_widget_tolerated` — source pode ser anexada a qualquer widget (LABEL, etc.); core não enforça tipo.

**Não testado em 2.2** (deferido para fatia aditiva futura):
- **LIST com 1M items** emit O(viewport) — exige rewriting de `capy_widget_emit_recursive` para consultar `virtual_source` ao invés de `children[]`.
- **Row recycling** — pool de widgets renderable rows reusados.
- **Determinismo end-to-end** com scroll position — idem (depende do emit rewiring).
- **Source com count variável** sem crash — idem (host renderer trata).

## Notas

- **Plumbing entregue, viewport emit deferido.** O plano original previa rewriting de `capy_widget_emit_recursive` para LIST/TREE/TABLE consultarem o data source no lugar de `children[]`. Isso é invasivo (toca o caminho de emit que tem byte-equivalence garantida desde 0.2) e seria melhor numa fatia futura dedicada. Por enquanto a infraestrutura está toda no lugar: descritor + ponteiro no widget + helpers de acesso + event type de prefetch.
- Caller é responsável por consistência da data source (sem cache interno, sem validação do count entre chamadas).
- Sem allocação interna; tudo passa por function pointers caller-fornecidos.
- Determinístico: dado `(source, index)`, o widget core nunca varia o valor retornado — é sempre o que `get_item` produz.
- `virtual_source` em widget non-LIST/TREE/TABLE é tolerado mas semanticamente neutro — o host renderer decide se honra.
- Performance: documenta tier 2 budget para listas virtualizadas. Benchmarks ficam para a fatia que entregar o viewport emit.

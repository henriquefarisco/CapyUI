## Objetivo

LIST, TREE e TABLE renderizam apenas itens visíveis (viewport). Suporte a milhões de itens sem regressão de frame budget.

## Escopo CapyUI

- Conceito de **virtual data source**:
  ```c
  typedef int (*capy_virtual_get_count_fn)(void *user_data);
  typedef int (*capy_virtual_get_item_fn)(void *user_data, uint32_t index,
                                          char *out_text, uint16_t cap);

  struct capy_virtual_data_source {
    capy_virtual_get_count_fn get_count;
    capy_virtual_get_item_fn get_item;
    void *user_data;
  };

  void capy_widget_set_virtual_source(struct capy_widget *w,
                                      const struct capy_virtual_data_source *src);
  ```
- LIST/TREE/TABLE consultam source apenas para items no viewport (calculado a partir de scroll position + bounds).
- Recycle de child widgets: pool fixo de renderable rows; reusados ao scrollar.
- Eventos: `CAPY_WIDGET_EVENT_VIRTUAL_REQUEST_RANGE(start, end)` permite host pre-fetch.

## Testes obrigatórios

1. **LIST com 1M items** emit é O(viewport).
2. **Scroll suave** reusa rows existentes.
3. **Lazy load** chamada apenas para items visíveis.
4. **Determinismo:** mesma data source + mesmo scroll → mesmo display-list.
5. **Source que muda count** atualiza sem crash.

## Notas

- Caller é responsável por consistência da data source.
- Sem cache interno de items; cada emit chama get_item para visíveis.
- Performance: dobra tier 2 budget para listas virtualizadas.
